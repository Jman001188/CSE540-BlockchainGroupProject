# APIs

## Authentication

### User Login

Description: Sends a request with a hashed password to authenticate the user. Responds with a session token and user and company information.

Endpoint
```sh
POST /auth/login
```
Request:
```sh
{
	"email": "name@company.com",
	"password": "hashedpassword="
}
```

Response:
```sh
{
	"sessionToken": "jwt-abc-xyz",
	"user": {
		"userID": 12355251234,
		"first name": "first",
		"last name": "last"
	}
	"company": {
		"companyID": 12312376823,
		"companyName": "Company Name",
		"permission": "Basic"
		"otherInformation": "..."	
	}
}
```

### User Logout

Endpoint
```sh
POST /auth/logout
```
Request:

### User Registration

Endpoint
```sh
POST /auth/register
```
Request:
```sh
{
	"type": "basic",
	"companyId": "company-uuid-456,
	"email": "user@company.com",
}
```

Response:
```sh
{
	"userId": "user-uuid-123",
	"registrationToken": ""
}
```

# Frontend Overview

## Onboarding Screen

Company accounts are created manually to prevent unauthorized registrations under existing or unowned companies. During this onboarding screen, an initial user account is established, acting as an administrator who can then add more administrators or basic users under the company.

Onboarding Data:
- Company Name
- User Email (used as the initial email with company admin privileges)
- User Password
- Public Key (Optional)
- Description (Optional)
- Optional additional fields

## Login Screen
- Fields
	- Login Field
	- Password Field

- Register Account Button
	- This button will open a new screen that has registration fields in it. The user will provide their desired login information to includes a registration token that is generated by a company admin for the specific user.

## Landing Page
Buttons:
- Register Batch
- Send Batch
- Receive Batch
- User Profile
- Company Profile (Admins Only)
- Company Users (Admins Only)

Note: Each of these options will send the user to a new screen. Treat this page as the navigation page.

## Register Batch Screen
Fields:
- Item Name
- Item Description
- Item Count
- Item Weight
- File Upload (optional)
- Optional additional data

Flow:
1. User enters batch information
2. User confirms submission
3. Meta Mask opens to sign the registration with the company private key
4. After signing, the contract’s register function is called

## Send Batch Screen
Flow:
1. Scan Item QR Code
2. Scan Recipient QR Code
	- This will include the user's account ID, or another ID that perhaps is generated on login, allowing only that user to receive the transfer request.
3. Confirm sending
4. Receiver approves the transfer on their device
5. Contract is called confirming transaction

  

## Receive Batch Screen
Pending Transfers View:
- Displays transfers intended for the company’s public key
- Filters by the user
- Once clicked, expands to a new view/screen with the information of the transfer request, allowing them to accept or deny the request.

Show QR Code:
- Contains the company public key and the user session ID
	- This is used to be able to filter transfer request on a by user basis to not flood many people in the same company with requests they don't need to see.
- Used by the sender to target this specific receiving session
